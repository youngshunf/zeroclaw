# 重构方案：多租户抽象层（消除上游合并冲突）

> 版本：1.0
> 日期：2026-03-22
> 作者：ZeroClaw / 顺富审核
> 状态：待审核

---

## 一、问题分析

### 当前状况

唤星在 17 个上游文件中注入了 **154 处** `#[cfg(feature = "huanxing")]` 条件编译。每次合并上游 master，这些散布在上游代码中的条件块频繁与上游改动冲突。

**注入分布统计：**

| 文件 | 注入次数 | 冲突风险 | 类型 |
|------|:---:|:---:|------|
| `channels/mod.rs` | 101 | 🔴极高 | 其中 66 处是测试代码 struct 初始化 |
| `gateway/ws.rs` | 17 | 🟡高 | per-user session + observer 隔离 |
| `config/schema.rs` | 11 | 🟡中 | Config struct 字段 + Default impl |
| `gateway/mod.rs` | 4 | 🟢低 | 路由 merge（符合规范） |
| `channels/napcat.rs` | 3 | 🟢低 | 语音处理 |
| `tools/{shell,file_*}.rs` | 各2 | 🟢低 | tenant security 获取 |
| `tools/mod.rs` | 2 | 🟢低 | 工具注册 |
| 其他 7 文件 | 各1 | 🟢低 | 模块声明/配置默认值 |

### 核心矛盾

`channels/mod.rs`（10370 行）是上游改动最频繁的文件，我们在里面有 101 处注入。**其中 66 处仅仅是因为 `ChannelRuntimeContext` 结构体加了 2 个 cfg 字段，导致每个测试函数都要写 2 行 cfg。**

剩余的 35 处业务逻辑注入可归类为 5 种模式：

| 模式 | 次数 | 说明 |
|------|:---:|------|
| 租户解析 | 11 | `tenant_router.resolve()` → 得到 TenantContext |
| 上下文覆盖 | 5 | model/provider/system_prompt/temperature 覆盖 |
| Memory 隔离 | 3 | `effective_memory` 指向 tenant memory |
| History 隔离 | 3 | `effective_histories` 指向 tenant HashMap |
| Session 隔离 | 2 | session backend 指向 tenant workspace |
| 工具沙箱 | 1 | `with_tenant_context()` 注入 task-local |
| Struct 字段 | 2 | `ChannelRuntimeContext` 的 2 个 cfg 字段 |
| Import/声明 | 8 | `use`/`mod` 声明 |

**本质上，这 5 种模式解决的是同一个问题：「根据消息来源确定运行时上下文」。**

---

## 二、设计目标

1. **上游文件零侵入**：`channels/mod.rs` 中的 101 处 cfg → **0 处**
2. **合并冲突最小化**：上游修改 channels/mod.rs 时不再与我们冲突
3. **功能完整保持**：所有多租户隔离功能不变
4. **向后兼容**：无 huanxing feature 时行为完全不变
5. **可测试**：抽象层本身可独立测试

---

## 三、方案设计

### 核心思路：Trait 抽象 + Hook 注入

定义一个 `trait MessageContextResolver`，将分散的 5 种注入模式统一为一次解析调用。上游代码只调用 trait 方法，不感知具体实现。

### 3.1 新增 Trait：`MessageContextResolver`

```rust
// src/channels/context_resolver.rs（新文件，属于上游代码）

use std::sync::Arc;
use std::path::PathBuf;
use crate::memory::Memory;
use crate::providers::ChatMessage;

/// 消息处理的运行时上下文，由 resolver 解析后返回。
/// 上游代码只依赖这个 struct，不关心它怎么生成的。
pub struct MessageContext {
    /// 系统提示词
    pub system_prompt: String,
    /// 模型名
    pub model: Option<String>,
    /// Provider 名
    pub provider: Option<String>,
    /// Temperature
    pub temperature: Option<f64>,
    /// API key（per-tenant 或全局）
    pub api_key: Option<String>,
    /// 向量记忆
    pub memory: Arc<dyn Memory>,
    /// 对话历史（per-session 隔离）
    pub conversation_histories: ConversationHistoryMap,
    /// 会话持久化后端
    pub session_manager: Option<Arc<dyn SessionBackend>>,
    /// 工作区目录
    pub workspace_dir: PathBuf,
    /// 安全策略
    pub security: Option<Arc<SecurityPolicy>>,
    /// Agent 标识（用于权限检查）
    pub agent_id: String,
    /// 是否为 guardian（未注册用户）
    pub is_guardian: bool,
    /// 用户显示名
    pub nickname: Option<String>,
    /// AI 角色名
    pub star_name: Option<String>,
}

/// 消息上下文解析器。
/// 上游提供默认实现（直接返回全局 context），
/// 唤星提供多租户实现（按 sender 路由到不同 TenantContext）。
pub trait MessageContextResolver: Send + Sync {
    /// 解析消息上下文。channel_name 如 "lark", sender_id 如 "ou_xxx"
    fn resolve(
        &self,
        channel_name: &str,
        sender_id: &str,
    ) -> impl std::future::Future<Output = MessageContext> + Send;

    /// 失效缓存（注册/配置变更后调用）
    fn invalidate(&self, _channel: &str, _sender_id: &str) {}

    /// 是否启用多租户模式
    fn is_multi_tenant(&self) -> bool { false }
}
```

### 3.2 上游默认实现

```rust
// src/channels/context_resolver.rs（同文件）

/// 默认解析器：直接使用全局配置，无多租户路由。
pub struct DefaultContextResolver {
    global_context: MessageContext,
}

impl MessageContextResolver for DefaultContextResolver {
    async fn resolve(&self, _channel: &str, _sender: &str) -> MessageContext {
        self.global_context.clone()
    }
}
```

### 3.3 唤星多租户实现

```rust
// src/huanxing/multi_tenant_resolver.rs（新文件，唤星目录内）

/// 多租户上下文解析器。
/// 包装 TenantRouter，将 resolve() 调用转换为 TenantContext 查找。
pub struct MultiTenantResolver {
    tenant_router: Arc<TenantRouter>,
    global_config: Arc<Config>,
}

impl MessageContextResolver for MultiTenantResolver {
    async fn resolve(&self, channel: &str, sender_id: &str) -> MessageContext {
        let tenant = self.tenant_router.resolve(channel, sender_id).await;
        MessageContext {
            system_prompt: tenant.system_prompt.clone(),
            model: tenant.model.clone(),
            provider: tenant.provider.clone(),
            temperature: tenant.temperature,
            api_key: tenant.api_key.clone(),
            memory: tenant.memory.clone(),
            conversation_histories: tenant.conversation_histories.clone(),
            session_manager: tenant.session_manager.clone(),
            workspace_dir: tenant.workspace_dir.clone(),
            security: tenant.security.clone(),
            agent_id: tenant.agent_id.clone(),
            is_guardian: tenant.is_guardian,
            nickname: tenant.nickname.clone(),
            star_name: tenant.star_name.clone(),
        }
    }

    fn invalidate(&self, channel: &str, sender_id: &str) {
        self.tenant_router.invalidate(channel, sender_id);
    }

    fn is_multi_tenant(&self) -> bool { true }
}
```

### 3.4 ChannelRuntimeContext 改造

**Before（当前）：**
```rust
pub struct ChannelRuntimeContext {
    // ... 20+ 个字段 ...
    #[cfg(feature = "huanxing")]
    tenant_router: Option<Arc<TenantRouter>>,
    #[cfg(feature = "huanxing")]
```

**After（重构后）：**
```rust
pub struct ChannelRuntimeContext {

```

**效果：测试代码中 66 处 cfg 全部消除。**

### 3.5 消息处理链改造

**Before（当前 channels/mod.rs，~35处 cfg 分支）：**
```rust
// 第2044行
#[cfg(feature = "huanxing")]
let tenant_ctx = if let Some(ref router) = ctx.tenant_router {
    router.resolve(&channel_name, &sender_id).await
} else { ... };
#[cfg(not(feature = "huanxing"))]
let tenant_ctx = ...;

// 第2196行
#[cfg(feature = "huanxing")]
let effective_model = tenant_ctx.model.clone().or(ctx.model.clone());
#[cfg(not(feature = "huanxing"))]
let effective_model = ctx.model.clone();

// 第2250行
#[cfg(feature = "huanxing")]
let effective_memory = tenant_ctx.memory.clone();
#[cfg(not(feature = "huanxing"))]
let effective_memory = ctx.memory.clone();

// ... 重复 N 次 ...
```

**After（重构后，0 处 cfg）：**
```rust
// 一次调用，得到完整上下文
let msg_ctx = ctx.context_resolver
    .resolve(&channel_name, &sender_id)
    .await;

// 直接使用，无需条件编译
let effective_model = msg_ctx.model.clone();
let effective_memory = msg_ctx.memory.clone();
let effective_histories = msg_ctx.conversation_histories.clone();
let effective_session = msg_ctx.session_manager.clone();
// ...
```

### 3.6 工具安全策略改造

**Before（4个工具文件各 2 处 cfg）：**
```rust
// src/tools/shell.rs
#[cfg(feature = "huanxing")]
let security = crate::huanxing::skill_market_tools::tenant_security()
    .unwrap_or_else(|| self.security.clone());
#[cfg(not(feature = "huanxing"))]
let security = self.security.clone();
```

**After（使用 task-local，对上游透明）：**
```rust
// src/tools/shell.rs — 改为统一接口，无 cfg
let security = crate::tools::get_active_security()
    .unwrap_or_else(|| self.security.clone());
```

```rust
// src/tools/mod.rs — 新增通用 task-local（非 huanxing 特有）
tokio::task_local! {
    static ACTIVE_SECURITY: Arc<SecurityPolicy>;
}

pub fn get_active_security() -> Option<Arc<SecurityPolicy>> {
    ACTIVE_SECURITY.try_with(|s| s.clone()).ok()
}
```

这个改动让上游也受益——**任何需要 per-request 安全策略覆盖的场景都可以用**，不限于唤星。

### 3.7 gateway/ws.rs 改造

同样的模式：将 per-user session backend 和 observer 收敛到 resolver：

```rust
// Before: 17 处 cfg
#[cfg(feature = "huanxing")]
session_backend: Option<Arc<dyn SessionBackend>>,
#[cfg(feature = "huanxing")]
ws_observer: Arc<WsObserver>,

// After: 通过 context_resolver 获取 per-user context
let msg_ctx = state.context_resolver.resolve("ws", &agent_name).await;
let session_backend = msg_ctx.session_manager;
```

### 3.8 config/schema.rs 的处理

Config struct 的 cfg 字段**保留不变**（11处），因为：
- 这些是 serde 反序列化字段，必须用 cfg gate
- Config struct 上游改动频率低
- 都是独立字段行，冲突概率极小

## 五、实施步骤

### Phase 1：定义抽象层（无行为变化）
1. 创建 `src/channels/context_resolver.rs`：定义 trait + MessageContext + DefaultResolver
2. 创建 `src/huanxing/multi_tenant_resolver.rs`：唤星多租户实现
3. `src/tools/mod.rs`：新增 `ACTIVE_SECURITY` task-local + `get_active_security()`

### Phase 2：改造 channels/mod.rs
1. `ChannelRuntimeContext` 去掉 2 个 cfg 字段，加 `context_resolver: Arc<dyn MessageContextResolver>`
2. 初始化时：无 huanxing feature → `DefaultContextResolver`；有 → `MultiTenantResolver`
3. 消息处理链：所有 `#[cfg(feature = "huanxing")]` 分支替换为 `ctx.context_resolver.resolve()` 调用
4. 测试代码：66 处 cfg 自动消除（struct 不再有 cfg 字段）

### Phase 3：改造 gateway/ws.rs + tools/*.rs
1. ws.rs：per-user session/observer 通过 resolver 获取
2. tools：4个文件的 `tenant_security()` → `get_active_security()`

### Phase 4：验证
1. `cargo build`（无 feature）通过
2. `cargo build --features huanxing` 通过
3. 功能测试：注册用户、多用户消息隔离、工具执行、定时任务
4. 模拟上游合并：从 upstream/master cherry-pick 最近 10 个改动 channels/mod.rs 的 commit，验证零冲突

## 六、风险评估

| 风险 | 概率 | 影响 | 缓解 |
|------|:---:|:---:|------|
| Trait object 性能开销 | 低 | 低 | 每消息一次 resolve，本身就有 DB 查询，vtable 开销可忽略 |
| MessageContext clone 开销 | 低 | 低 | 内部全是 Arc/Option，clone 只是引用计数 +1 |
| 上游后续改动 trait 不兼容 | 中 | 中 | MessageContext 字段从上游 ChannelRuntimeContext 派生，上游加字段我们对应加 |
| 重构过程引入 bug | 中 | 高 | 分 Phase 推进，每 Phase 编译+测试，保留 git tag 可回滚 |

---

## 七、预期收益

1. **合并效率**：上游同步从 2-4 小时降到 10-30 分钟
2. **代码清晰度**：channels/mod.rs 减少 ~200 行条件编译噪音
3. **可维护性**：新增多租户逻辑只需改 `multi_tenant_resolver.rs`，不碰上游
4. **可测试性**：resolver 可 mock，单元测试无需真实 DB
5. **上游贡献可能**：`MessageContextResolver` trait 对上游也有价值（插件化 context 解析）

## 附录 B：保留 cfg 的文件清单（无需重构）

这些文件的 cfg 注入数量少（1-11处）、冲突风险低，且属于以下类型无法避免：
- **serde 字段声明**：`config/schema.rs` 的 11 处，必须用 cfg gate
- **模块声明**：`main.rs`/`lib.rs` 各 1 处 `mod huanxing`
- **渠道特有逻辑**：`napcat.rs`/`lark.rs`/`tts.rs` 的语音处理
- **路由注册**：`gateway/mod.rs` 的 merge，已符合开发规范
- **启动逻辑**：`daemon/mod.rs` 的 heartbeat 启动

合计 26 处，均为低冲突风险。
