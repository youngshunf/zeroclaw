# 唤星 ZeroClaw 开发规范

> 版本：2.1
> 更新：2026-03-25
> 适用范围：huanxing-zeroclaw 仓库所有开发者

## 核心原则

**一句话总结：上游文件用 trait 抽象注入，所有唤星逻辑都在独立目录。**

```
┌─────────────────────────────────────────────────┐
│ Layer 4: 唤星桌面端 (clients/desktop/)           │ ← 纯唤星代码，零冲突
│   Tauri + React + Sidecar 管理                   │
├─────────────────────────────────────────────────┤
│ Layer 3: 唤星 Rust 扩展层 (src/huanxing/)        │ ← feature-gated 独立模块
│   TenantRouter + TenantContext + 业务工具         │
├─────────────────────────────────────────────────┤
│ Layer 2: 抽象层 (src/channels/context_resolver)  │ ← 上游 trait 定义（通用）
│   MessageContextResolver + DefaultResolver       │
├─────────────────────────────────────────────────┤
│ Layer 1: ZeroClaw Core (src/*)                   │ ← 不改，保持与上游同步
│   Agent + Providers + Tools + Gateway            │
└─────────────────────────────────────────────────┘
```

---

## 与上游的关系

### 上游合并策略

唤星是 ZeroClaw 的 fork，通过 `git remote upstream` 同步上游更新。

**目标：上游怎么改 channels/mod.rs 都不跟我们冲突。**

### 隔离边界

| 层级 | 文件位置 | 上游感知 | 合并冲突风险 |
|------|---------|:---:|:---:|
| 唤星逻辑 | `src/huanxing/` | 无 | 零 |
| 桌面端 | `clients/desktop/` | 无 | 零 |
| 抽象层 | `src/channels/context_resolver.rs` | 有（通用 trait） | 极低 |
| 配置字段 | `src/config/schema.rs` | 有（cfg gate） | 低 |
| 入口声明 | `main.rs`/`lib.rs` | 有（1行 mod） | 极低 |

## 代码修改规范

### 规则 1：上游核心文件禁止插 `#[cfg(feature = "huanxing")]`

**重灾区文件（绝对禁止）：**
- `src/channels/mod.rs`
- `src/gateway/ws.rs`
- `src/tools/*.rs`

**唯一例外（允许少量 cfg）：**
- `src/config/schema.rs` — serde 字段声明（无法避免）
- `src/main.rs` / `src/lib.rs` — 1 行模块声明
- `src/gateway/mod.rs` — 路由 merge（已规范化）
- `src/daemon/mod.rs` — 启动入口（1 处）
- `src/channels/{napcat,lark,tts}.rs` — 渠道特有逻辑（少量）

### 规则 2：多租户逻辑通过 Trait 抽象注入

**正确（trait 注入）：**
```rust
// channels/mod.rs — 一次 resolve 调用
let msg_ctx = ctx.context_resolver.resolve(&channel, &sender_id).await;
let model = msg_ctx.model;
let memory = msg_ctx.memory;
let prompt = msg_ctx.system_prompt;
// 无 cfg，上游完全透明
```

**禁止（内联条件编译）：**
```rust
// ❌ 不要在 channels/mod.rs 里写这种
#[cfg(feature = "huanxing")]
let model = tenant_ctx.model.clone().or(ctx.model.clone());
#[cfg(not(feature = "huanxing"))]
let model = ctx.model.clone();
// 重复 N 次 = N 个冲突点
```

### 规则 3：工具安全策略用 task-local

**正确：**
```rust
// tools/shell.rs — 统一接口
let security = crate::tools::get_active_security()
    .unwrap_or_else(|| self.security.clone());
```

**禁止：**
```rust
// ❌ 不要在工具文件里写 cfg
#[cfg(feature = "huanxing")]
let security = crate::huanxing::skill_market_tools::tenant_security()...;
#[cfg(not(feature = "huanxing"))]
let security = self.security.clone();
```

### 规则 4：复杂模块抽离为独立实现（如 WebSocket, Tools）

**WebSocket 隔离（核心经验）：**
- 禁止修改上游 `src/gateway/ws.rs`（这是合并冲突的重灾区）。
- 唤星自定义的多租户、多 Session 实时会话 WS 应单独实现在 `src/huanxing/hx_ws.rs`。
- 在 `src/gateway/mod.rs` 路由入口处通过 `cfg` 挂载独立路由。

**工具注册隔离：**
- 禁止在上游 `src/tools/mod.rs` 中直接实例化唤星工具。
- 唤星的所有 API 依赖工具必须在 `src/huanxing/tools.rs` 中定义，并在 `src/huanxing/register.rs` 中统一注册。
- 在 `src/tools/mod.rs` 只需调用一次 feature-gated 的 `huanxing_all_tools()` 方法。

### 规则 4：路由用 merge，不逐行添加

**允许（gateway/mod.rs 只加一行）：**
```rust
#[cfg(feature = "huanxing")]
let app = app.merge(crate::huanxing::router::huanxing_routes(shared_state.clone()));
```

**禁止：**
```rust
// ❌ 不要逐个注册路由
.route("/api/agents", get(huanxing::agents::list))
.route("/api/sessions", get(huanxing::sessions::list))
```

### 规则 5：配置用独立 section

```toml
# ZeroClaw 原版配置
[llm]
provider = "openai_compat"

# 唤星扩展配置（独立 section）
[huanxing]
enabled = true
api_base_url = "https://api.huanxing.dcfuture.cn"
```

### 规则 6：State 用 Extension，不改 AppState

```rust
// ✅ 用 axum Extension 注入
#[cfg(feature = "huanxing")]
let app = app.layer(Extension(HuanxingState { ... }));

// ❌ 不在 AppState 加字段
```

## 新功能开发流程

### 开发多租户相关功能时

1. **实现放 `src/huanxing/`** — 不碰上游代码
2. **如需在消息处理链中注入行为** — 扩展 `MessageContext` struct 的字段，在 `MultiTenantResolver::resolve()` 中填充
3. **如需新工具** — 在 `src/huanxing/tools.rs` 注册，权限在 `permissions.rs` 配置
4. **如需新渠道特有逻辑** — 只在对应渠道文件加少量 cfg（如 `napcat.rs` 语音）
5. **如需新 API** — 在 `src/huanxing/` 定义路由，通过 `gateway/mod.rs` 的 merge 一行接入

### 开发检查清单

提交前验证：
- [ ] `cargo build`（无 feature）通过
- [ ] `cargo build --features huanxing` 通过
- [ ] 没有在 `channels/mod.rs` 加 `#[cfg(feature = "huanxing")]`
- [ ] 没有在 `gateway/ws.rs` 加 `#[cfg(feature = "huanxing")]`
- [ ] 没有在 `tools/*.rs` 加 `#[cfg(feature = "huanxing")]`
- [ ] 新逻辑在 `src/huanxing/` 目录内
- [ ] 桌面端代码在 `clients/desktop/`

## 桌面端开发规范

### 前端独立原则

桌面端 `clients/desktop/` 完全独立于 `web/` 目录：

| 配置项 | 指向 |
|--------|------|
| vite alias `@` | `./src`（不是 `../../web/src`） |
| tsconfig paths | `./src/*` |
| Tailwind @source | `.` |
| node_modules | 独立安装 |

### 复用上游组件

需要复用 web 组件时，**复制到桌面端目录**：

```bash
# ✅ 复制
cp web/src/components/ChatMessage.tsx clients/desktop/src/components/

# ❌ 跨目录 import
import ChatMessage from "../../../web/src/components/ChatMessage"
```

### Tauri Sidecar 架构

```
用户登录 → 获取 llm_token → Tauri 创建 sidecar → sidecar 使用独立配置/端口
```

- 配置文件：`~/.huanxing/config.toml`
- 端口：动态分配
- 生命周期：跟随 Tauri 窗口

## Git 工作流

### 分支策略

| 分支 | 用途 |
|------|------|
| `main` | 稳定版本 |
| `huanxing-clean` | 上游合并工作分支 |
| `feature/*` | 功能开发 |

### 上游同步流程与合并策略（核心经验）

唤星代码与上游主干合并时，采取 **“零入侵隔离设计 + 手动精细化合并”** 策略：

```bash
# 1. 获取上游
git fetch upstream
git checkout huanxing-clean
git merge upstream/master

# 2. 解决冲突（按策略进行）
# 3. cargo check && cargo check --features huanxing
# 4. cargo test --test huanxing_feature_coverage --features huanxing
# 5. 合并到 main
```

#### 常见冲突解决策略：

1. **结构化更改优先上游**：如果上游修改了控制流（如增加流式回退 `Err(stream_err)`、`step_timeout_secs`、返回元组扩充为 5/7-tuple），**无条件接受上游的结构**，然后手动将我们的特有逻辑（如 Debug Log、解析异常探测）注入到对应的位置。
2. **`Cargo.toml`**：同时保留上游新增的 feature（如 webauthn）和我们自己的 workspace members/features。
3. **`config/schema.rs`**：必定有冲突。同时保留唤星的独有安全字段（`syscall_anomaly`、`semantic_guard` 等）和上游新增字段，并更新 `Default` 实现。
4. **`channels/mod.rs`（最复杂冲突区）**：
   - 管道增强：同时保留双方的新管道（如我们的图像本地化 `localize_remote_image_markers` 和上游的 `media_pipeline`）。
   - 构造参数变更：跟进上游 Channel 构造函数的变化（如 `new_with_proxy`），补充我们的特有参数。
   - `#[cfg(feature = "huanxing")]` 注册区：在上下文中寻找合适位置，确保不要被覆盖（如 `register_live_channels`、`context_resolver` 注入）。
5. **依赖同步**：合并后注意结构体新增字段（如 `ChannelMessage` 增加 `attachments`），需手动修复 `napcat.rs` 和 `wechat_pad.rs` 报错。

**预期冲突点：** `config/schema.rs`, `channels/mod.rs`, `Cargo.toml`, core agent loop (`loop_.rs`)。如果是 `tools/mod.rs` 或 `ws.rs` 大面积冲突，说明代码隔离度退化，需要立刻重构回 `src/huanxing/`。

## 常见问题

### Q: 需要修改上游的某个函数签名怎么办？

优先级：
1. 用 wrapper 函数包装
2. 用 trait extension 扩展
3. 提 PR 给上游
4. 实在不行直接改，标注 `// HUANXING: 修改原因`

### Q: MessageContext 需要加新字段怎么办？

1. 在 `src/channels/context_resolver.rs` 的 `MessageContext` 加字段（给个 Default）
2. 在 `src/huanxing/multi_tenant_resolver.rs` 的 `resolve()` 中填充
3. DefaultResolver 使用 Default 值
4. 在 channels/mod.rs 消费时直接 `msg_ctx.new_field`

### Q: 上游更新后编译失败怎么办？

1. 上游 API 签名变化 → 更新 `src/huanxing/` 中的调用
2. Config struct 变化 → 检查 `HuanXingConfig`
3. ChannelRuntimeContext 变化 → 检查 `MessageContext` 是否需要同步
4. 依赖版本冲突 → 检查 `Cargo.toml`
