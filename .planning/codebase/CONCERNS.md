# 技术债务与关注点

**分析日期：** 2026-03-21

## 架构层面

### 唤星与上游同步维护

**问题：** 唤星扩展基于 ZeroClaw fork，需要持续同步上游更新

**影响：**
- 上游 API 变更可能破坏唤星扩展
- 手动合并冲突耗时
- 测试覆盖不足以捕获所有兼容性问题

**当前缓解措施：**
- Feature gate (`#[cfg(feature = "huanxing")]`) 隔离唤星代码
- 上游文件只添加最小 cfg 注解
- 独立 `huanxing-clean` 分支用于上游合并

**建议：**
- 增加集成测试覆盖唤星特有路径
- 自动化上游同步 CI 检查
- 考虑将唤星扩展提取为独立插件（长期）

---

### 桌面端与 web 目录耦合

**问题：** 桌面端 `clients/desktop/` 曾依赖 `web/` 目录，现已解耦但遗留配置

**影响：**
- 历史 import 路径可能残留
- 构建配置复杂度高
- 新开发者容易误用 web 组件

**当前状态：**
- 桌面端已完全独立（vite alias `@` → `./src`）
- `web/` 目录保留用于 ZeroClaw 原版 Web UI

**建议：**
- 清理所有跨目录 import（已完成）
- 文档明确说明两者独立性（已完成）

---

### 多租户路由缓存一致性

**问题：** `TenantRouter` 使用内存缓存（`RwLock<HashMap>`），数据库更新后缓存可能过期

**影响：**
- 用户信息更新后需手动调用 `invalidate_cache`
- 缓存未命中时查询数据库，性能抖动
- 多实例部署时缓存不同步

**当前缓解措施：**
- 提供 `hx_invalidate_cache` 工具手动清理
- 缓存 TTL 机制（未实现）

**建议：**
- 实现基于时间的缓存过期（TTL）
- 考虑使用 Redis 作为共享缓存层（多实例场景）
- 数据库更新时自动触发缓存失效

---

## 代码质量

### 错误处理粒度不一致

**问题：** 部分模块使用 `anyhow::Result`，部分使用 `thiserror` 自定义错误

**影响：**
- 错误类型不统一，难以分类处理
- 上层调用者无法精确匹配错误类型
- 日志中错误信息可能不够结构化

**位置：**
- `src/huanxing/` 大量使用 `anyhow::bail!`
- `src/providers/` 部分使用自定义 `ProviderError`

**建议：**
- 定义唤星专属错误类型（`HuanxingError`）
- 关键路径（认证、支付）使用结构化错误
- 保留 `anyhow` 用于快速原型和工具层

---

### 测试覆盖率不足

**问题：** 唤星扩展层（`src/huanxing/`）缺少单元测试和集成测试

**影响：**
- 重构风险高
- 回归 bug 难以发现
- 新功能可能破坏现有逻辑

**当前状态：**
- ZeroClaw 核心有完整测试套件（`tests/`）
- 唤星层几乎无测试（仅少量内联测试）

**建议：**
- 为 `TenantRouter` 添加单元测试（缓存逻辑、路由逻辑）
- 为 `ApiClient` 添加 mock 测试（HTTP 调用）
- 为唤星工具（`hx_register_user` 等）添加集成测试

---

### 硬编码配置

**问题：** 部分配置硬编码在代码中，未通过配置文件暴露

**位置：**
- `src/huanxing/api_client.rs` 中的 API 端点 URL
- `src/huanxing/tools.rs` 中的超时时间
- `src/gateway/mod.rs` 中的速率限制常量

**影响：**
- 环境切换（开发/生产）需要重新编译
- 无法动态调整参数
- 测试时难以 mock

**建议：**
- 将所有端点 URL 移到 `HuanxingExtConfig`
- 超时、重试次数等参数配置化
- 使用环境变量覆盖机制

---

## 性能

### SQLite 并发写入瓶颈

**问题：** SQLite 单写入者限制，高并发场景下可能成为瓶颈

**影响：**
- 多租户同时写入记忆时排队
- 会话历史写入延迟
- 租户数据库（`data/users.db`）锁竞争

**当前缓解措施：**
- 使用 WAL 模式（Write-Ahead Logging）
- 连接池复用

**建议：**
- 考虑 PostgreSQL 作为生产环境后端
- 实现写入批处理（batch insert）
- 分片租户数据库（按用户 ID 哈希）

---

### 内存缓存无限增长

**问题：** `TenantRouter` 缓存无上限，长时间运行可能内存泄漏

**影响：**
- 租户数量增长时内存占用线性增长
- 无 LRU 淘汰机制

**建议：**
- 实现 LRU 缓存（使用 `lru` crate）
- 设置缓存大小上限（如 1000 个租户）
- 定期清理不活跃租户缓存

---

### Agent 工作区模板复制开销

**问题：** 每次创建 Agent 时完整复制模板目录（`templates/huanxing/_base_desktop/`）

**影响：**
- 创建 Agent 耗时（尤其模板包含大文件时）
- 磁盘空间占用（每个 Agent 独立副本）

**建议：**
- 使用符号链接共享只读文件（SOUL.md, config.toml）
- 仅复制需要修改的文件
- 考虑模板懒加载（首次使用时才复制）

---

## 安全

### Agent Key 明文存储

**问题：** Agent Key（`AGENT_SECRET_KEY`）存储在 `.env` 文件中，未加密

**影响：**
- 文件泄漏即泄漏所有 Agent 权限
- 无密钥轮换机制

**建议：**
- 使用系统密钥环（macOS Keychain, Linux Secret Service）
- 实现密钥轮换 API
- 支持多个 Agent Key（按用途隔离）

---

### 短信验证码存储

**问题：** 短信验证码存储在 SQLite（`data/users.db`），未加密

**影响：**
- 数据库泄漏即泄漏所有验证码
- 验证码有效期管理依赖应用层

**建议：**
- 验证码加密存储（ChaCha20Poly1305）
- 数据库级别的 TTL（或定期清理过期记录）
- 考虑使用 Redis 存储验证码（自动过期）

---

### CORS 配置过于宽松

**问题：** 部分环境 CORS 配置为 `allow_origins: ["*"]`

**影响：**
- 任意域名可调用 API
- CSRF 攻击风险

**建议：**
- 生产环境明确指定允许的域名
- 使用 `allow_credentials=True` 时必须指定具体域名
- 实现 CSRF Token 机制

---

## 可维护性

### 文档与代码不同步

**问题：** 部分文档（`docs/`）描述的 API 与实际实现不一致

**影响：**
- 新开发者误用 API
- 集成测试失败

**位置：**
- `docs/桌面端/Agent管理与会话管理方案v3.md` 中的 API 端点
- `CLAUDE.md` 中的配置示例

**建议：**
- 使用 OpenAPI/Swagger 自动生成 API 文档
- CI 检查文档中的代码示例是否可编译
- 定期审查文档与代码一致性

---

### 日志级别不一致

**问题：** 部分模块使用 `tracing::info!` 记录调试信息，部分使用 `tracing::debug!`

**影响：**
- 生产环境日志噪音
- 调试时难以过滤有用信息

**建议：**
- 统一日志级别规范：
  - `error!` — 严重错误（需要人工介入）
  - `warn!` — 可恢复异常（自动重试、降级）
  - `info!` — 正常生命周期事件（启动、关闭、连接）
  - `debug!` — 详细内部状态（仅调试时开启）
  - `trace!` — 极详细跟踪（性能分析）

---

### 配置文件 Schema 验证不足

**问题：** TOML 配置文件解析失败时错误信息不友好

**影响：**
- 用户配置错误时难以定位问题
- 缺少字段时使用默认值，可能导致意外行为

**建议：**
- 使用 `serde_ignored` 检测未知字段
- 提供配置验证命令（`zeroclaw config validate`）
- 生成 JSON Schema 供编辑器自动补全

---

## 已知 Bug

### Sidecar 进程僵尸

**问题：** Tauri 应用退出时，zeroclaw sidecar 进程可能未正确终止

**影响：**
- 端口占用（42620）
- 资源泄漏

**当前缓解措施：**
- `SidecarManager::adopt_existing()` 检测并复用现有进程
- 应用退出时发送 SIGTERM

**建议：**
- 实现进程心跳检测
- 使用 PID 文件锁防止多实例
- 应用崩溃时自动清理 sidecar

---

### WebSocket 重连风暴

**问题：** 渠道 WebSocket 断开后立即重连，可能触发速率限制

**影响：**
- 被平台封禁
- 日志刷屏

**当前缓解措施：**
- 部分渠道实现指数退避（Discord, Slack）
- 部分渠道未实现（Lark, DingTalk）

**建议：**
- 统一重连策略（指数退避 + 最大重试次数）
- 实现断路器模式（连续失败后暂停重连）
- 记录重连次数到可观测性指标

---

### 桌面端前端状态不同步

**问题：** 前端 React 状态与 sidecar 状态可能不一致（如 Agent 列表）

**影响：**
- UI 显示过期数据
- 用户操作失败（如删除已删除的 Agent）

**建议：**
- 实现 WebSocket 推送机制（sidecar → 前端）
- 前端定期轮询刷新（fallback）
- 使用乐观更新 + 回滚机制
